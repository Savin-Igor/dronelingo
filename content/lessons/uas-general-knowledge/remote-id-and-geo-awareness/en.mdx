## Two different systems

**Direct Remote ID** is outward-facing. It broadcasts operational data so authorities and other authorised receivers can identify the aircraft and operator in real time.

**Geo-awareness** is inward-facing. It helps the pilot and aircraft understand nearby UAS geographical zones and alerts about restrictions.

They solve different problems:

- Remote ID answers: *who is this drone and where is it now?*
- Geo-awareness answers: *what restrictions exist around this position?*

## Which classes depend on these features?

In the Open-category ecosystem, Remote ID and geo-awareness are associated especially with **C1, C2, and C3** class requirements. The exact technical implementation is the manufacturer’s job, but the operational consequence is the pilot’s job:

- confirm the system is enabled and healthy
- keep the dataset current when the manufacturer requires updates
- never assume the aircraft feature overrides the legal map source

For Latvia, the legal/geographical check still points back to `airspace.lv/drones` and the official zone data chain behind it.

## What pilots should do before launch

Treat Remote ID and geo-awareness as part of the pre-flight digital check:

1. confirm no system warning is active
2. confirm the app and aircraft location agree with reality
3. confirm the geographical-zone dataset is current if your system uses downloadable data
4. independently verify the location on `airspace.lv/drones`

If the aircraft says “all clear” but the official map says the zone is restricted, the official map wins.

<Scenario title="Your aircraft app shows no warning and the geofence view looks clear. On `airspace.lv/drones`, the same location is shown as a restricted UAS geographical zone that requires BGKIS. Which source controls your decision?">
`airspace.lv/drones` controls the decision. Geo-awareness is a safety aid, not the legal source of truth. If the aircraft dataset is stale or incomplete, it can fail silently.

The correct action is to follow the official zone source, submit BGKIS if required, and fly only after approval.
</Scenario>

<CommonMistakes>
- Treating Remote ID as if it were a “license to enter” restricted airspace.
- Assuming geo-awareness data in the app is always current.
- Ignoring official zone maps because the controller screen shows no warning.
- Turning a technical feature into a legal decision-maker instead of a supporting tool.
</CommonMistakes>
